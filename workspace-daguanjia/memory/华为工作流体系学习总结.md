## 一、核心工作流（9 个）- 必须精通

### 1.1 IPD 集成产品开发工作流 v6.6

**核心定位**: 从概念到发布的产品开发全流程

**9 个 Gate 门控**:
| Gate | 阶段 | 核心目标 | 关键产出 |
|------|------|---------|---------|
| Gate0 | 概念筛选 | 市场机会验证 | 市场分析报告 |
| Gate1 | 商业分析 | 商业模式验证 | 商业分析报告 |
| Gate2 | 用户验证 | 用户需求验证 | 用户研究报告 |
| Gate3 | 需求分析 | 需求明确化 | 7 个文档（概述/角色/场景/功能/页面/接口/数据） |
| Gate4 | 产品设计 | 方案明确化 | 9 个文档（概述/角色/页面流程/功能 BDD/页面 BDD/技术方案/风险/验收/里程碑） |
| Gate5 | UI 设计 | 视觉明确化 | 设计规范 + 各端 UI 设计图 |
| Gate6 | 研发编码 | 功能实现 | 代码 + 接口文档 |
| Gate7 | 测试验收 | 质量保证 | 测试报告 |
| Gate8 | 发布上线 | 上线运营 | 发布文档 + 运营计划 |

**v6.6 核心改进**:
- ✅ 质量门禁机制 - 每个 Gate 结束时的质量检查
- ✅ 阻断规则 - 门禁不通过不能进入下一阶段
- ✅ 红线检查 - 资金/安全/合规/性能红线
- ✅ Agent 分工 - 每个 Gate 明确 Agent 职责
- ✅ 自动化机制 - Agent 自动调用、自动校验
- ✅ 协同工作流 - 与其他工作流的协同关系

**产出物标准**:
- Gate3: 7 个文档（覆盖率 100%）
- Gate4: 9 个文档（覆盖率 100%）
- Gate5: 设计规范 + APP/H5/PC/工厂端 UI 设计图
- 覆盖率要求：场景→功能→页面→接口→数据表 100% 覆盖

**应用场景**: 防水智能报价工具开发（当前项目）

### 1.2 DSTE 战略到执行工作流 v13.4

**核心定位**: 从战略规划到执行监控的闭环管理

**6 个阶段**:
- G1 战略分析 → G2 目标设定 → G3 战略分解 → G4 资源配置 → G5 执行监控 → G6 复盘优化

**核心内容**:
- 战略分析：市场分析、竞争分析、能力分析、机会威胁
- 战略目标：目标描述、衡量指标、时间节点、责任人
- 战略举措：举措内容、优先级、资源需求、预期效果
- 战略解码：目标分解、部门目标、责任人、考核
- OKR：O+KR、进度、评分
- 执行计划：计划内容、资源、时间、里程碑

**产出物标准**:
| 产出物 | 最少字数 | 必须章节 |
|--------|---------|---------|
| 战略分析 | 2000 字 | 市场分析、竞争分析、能力分析、机会威胁 |
| 战略目标 | 1500 字 | 目标描述、衡量指标、时间节点、责任人 |
| 战略举措 | 1500 字 | 举措内容、优先级、资源需求、预期效果 |
| 战略解码 | 2000 字 | 目标分解、部门目标、责任人、考核 |

**质量门禁**: 每个阶段结束前必须质量检查，不通过不能进入下一阶段

**应用场景**: 项目战略规划、年度计划制定

### 1.3 LTC 线索到现金工作流 v13.3

**核心定位**: 从线索获取到回款的销售全流程

**6 个 Gate**:
- G1 线索获取 → G2 商机转化 → G3 方案报价 → G4 合同签订 → G5 交付回款 → G6 复盘优化

**销售策略三要素**（前置层）:
1. **客户定位** - 谁是我们的目标客户
2. **商机取舍** - 优先跟进哪些商机
3. **资源配称** - 销售资源怎么分配

**产出物标准**:
| 产出物 | 最少字数 | 必须章节 |
|--------|---------|---------|
| 线索清单 | 按需 | 线索来源、联系方式、评分、优先级 |
| 客户画像 | 每画像 500 字 | 基本信息、需求痛点、决策链、预算 |
| 商机确认书 | 1000 字 | 客户需求、预算、决策人、时间节点 |
| 解决方案 | 2000 字 | 客户需求、方案设计、价值、报价 |
| 合同 | 按需 | 合同条款、付款条件、交付物、验收标准 |

**日监控指标**:
- 赢单率 ≥ 目标值（报警线<目标值 -10%）
- 回款准时率 ≥ 90%（报警线<80%）
- 续费率 ≥ 85%（报警线<75%）

**应用场景**: 获客转化流程、销售管理

### 1.8 ITR 问题到解决工作流 v13.3

**核心定位**: 从问题发现到根本解决的系统性方法

**6 个阶段**:
- G1 问题发现 → G2 分拣 → G3 诊断 → G4 修复 → G5 验证 → G6 复盘

**问题管理三要素**:
1. **问题定位** - 问题分类、根因定位、影响评估
2. **优先级排序** - keep/hold/cut 决策、优先级排序
3. **资源配称** - 资源配置、里程碑承诺

**产出物标准**:
| 产出物 | 最少字数 | 必须章节 |
|--------|---------|---------|
| 问题登记 | 500 字 | 问题描述、影响范围、紧急程度 |
| 原因分析 | 1000 字 | 5Why 分析、鱼骨图、根因确定 |
| 解决方案 | 1000 字 | 对策、计划、资源、时间、责任人 |
| 验证确认 | 500 字 | 验证方法、结果、确认时间 |
| 复盘报告 | 1000 字 | 根因分析、措施、预防、效果 |

**日监控指标**:
- MTTR（平均修复时间）
- MTTS（平均解决时间）
- 解决率 ≥ 90%
- 重复率 ≤ 5%

**应用场景**: 售后问题处理、客户投诉处理

### 2.1 多 Agent 协作工作流 v8.0

**15 个 Agent 角色**:
| Agent | 角色 | 职责 | 审查谁 |
|-------|------|------|--------|
| main | Orchestrator | 任务路由、状态追踪 | - |
| coder | Builder | 代码开发 | tester |
| pm | Builder | 产品设计 | supervisor |
| designer | Builder | UI 设计 | supervisor |
| flash | Builder | 内容创作 | pm |
| researcher | Builder | 市场调研 | pm |
| data | Builder | 数据分析 | supervisor |
| tester | Reviewer | 测试、质量保证 | - |
| supervisor | Reviewer | 质量监督 | - |
| legal | Reviewer | 法务合规 | - |
| operator | Ops | 运营管理 | supervisor |
| customer | Ops | 客户服务 | supervisor |
| finance | Ops | 财务预算 | legal |
| hr | Ops | 人力资源 | supervisor |
| supply | Builder | 供应链 | supervisor |

**任务状态机**:
```
Inbox → Assigned → In Progress → Review → Confirm → Done | Failed
                              ↑
                        用户确认点（必须）
```

**交接协议**（固化模版）:
```
📦 交接消息
├─ 1. 完成了什么：[摘要]
├─ 2. 产物在哪：[精确路径]
├─ 3. 产出物清单：[路径 + 具体内容]
├─ 4. 如何验证：[测试命令/验收标准]
├─ 5. 已知问题：[未完成/风险点]
├─ 6. 下一步：[接收 Agent 该做什么]
└─ 7. 覆盖率：[产出物覆盖率%]
```

**自动调用机制**:
- Gate3 → pm, coder
- Gate4 → designer, pm
- Gate5 → designer, flash
- Gate6 → coder, tester, ops
- Gate7 → tester, supervisor
- Gate8 → operator, customer

**应用场景**: 团队协作、多 Agent 并行执行

### 2.5 工作流协同网络机制 v3.0

**核心功能**: 工作流之间的自动触发和协同

**自动触发规则**:
- 当任何工作流到达特定阶段时 → 自动触发协同工作流 → 自动调用对应 Agent

**触发器设计**:
```yaml
trigger_id: TRIGGER-IPD-GATE6-001
trigger_name: IPD Gate6 进入触发
trigger_timing: stage_enter
trigger_condition:
  workflow: IPD
  stage: Gate6
  gate_result: pass
trigger_actions:
  - action: call_workflow
    target: ITR
  - action: call_workflow
    target: PQA
  - action: call_agent
    target: [coder, tester, ops]
  - action: record_mir
```

**监控指标**:
- 触发成功率 100%（报警线<90%）
- 触发及时率 100%（报警线<80%）
- Agent 调用成功率 100%（报警线<85%）
- 协同完成率 ≥ 95%（报警线<80%）

**应用场景**: 工作流联动、跨流程协同

## 四、通用机制（所有工作流）

### 4.1 质量门禁机制

**核心原则**:
- 前置检查：每个阶段结束前必须质量检查
- 阻断机制：不通过不能进入下一阶段
- 问题归零：问题必须在当前阶段解决

**通用门禁项**:
| 检查项 | 通过标准 |
|--------|---------|
| 产出物完整性 | 100% 齐全 |
| 数据支撑性 | 有据可查 |
| 一致性 | 与上下游对应 |

**红线门禁**:
- 资金红线：涉及合同、回款
- 合规红线：涉及合同条款
- 服务红线：涉及服务交付
- 时效红线：涉及交付时效

### 4.2 MIR 同步协议

**强制字段**:
```yaml
task_id: string
```

- 每日 20:00：输出日报

### 4.3 产出物标准

**具体化要求**:
| 维度 | ❌ 不具体 | ✅ 具体 |
|------|---------|--------|

| 任务 | "完成任务" | "完成 X 任务，输出 Y 文档，验收标准 Z" |
| 客户 | "目标客户是中小企业" | "制造业、年营收 500-5000 万、IT 预算 50-200 万" |

**覆盖率要求**:
- 需求→设计：100% 覆盖
- 设计→代码：100% 覆盖
- 代码→测试：100% 覆盖
- 测试→验收：100% 覆盖

## 六、学习总结

### 6.1 核心收获

1. **IPD 是产品开发的核心流程** - 9 个 Gate 门控，每个 Gate 有明确的产出物标准
2. **质量门禁是关键机制** - 每个阶段结束前必须检查，不通过不能进入下一阶段
3. **产出物具体化是执行保障** - 不能笼统描述，必须具体到可直接执行
4. **覆盖率 100% 是质量门槛** - 上下游产出物必须 100% 对应
5. **MIR 同步是信息对齐机制** - 每 30 分钟同步，确保信息不丢失
6. **Agent 自动调用是效率保障** - 阶段进入时自动触发协同工作流和 Agent

### 6.2 应用到防水项目

1. **按 IPD 流程执行** - 当前在 Gate3→Gate4→Gate5，确保每个 Gate 产出物完整
2. **质量门禁检查** - 每个 Gate 结束时进行质量检查，不通过不进入下一阶段
3. **产出物具体化** - PRD、UI 设计稿等必须具体到可直接执行
4. **Agent 分工** - 小马负责产品、小爱负责设计、黄大仙负责测试
5. **MIR 同步** - 每 30 分钟同步项目状态，每日 20:00 输出日报
