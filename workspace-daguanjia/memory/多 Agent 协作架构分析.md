## 📊 当前架构

### 配置分析

**当前 openclaw.json 配置**：
```json
"agents": {
  "list": [
    {"id": "main"},
    {"id": "erkang", "workspace": "C:\\Users\\pony\\clawd-erkang"},
    {"id": "xiaoma", "workspace": "C:\\Users\\pony\\clawd-xiaoma"}
  ]
}
```

**绑定关系**：
| Agent | 飞书账号 | 工作空间 | Gateway |
|-------|----------|----------|---------|
| main | cli_a907cd5d34789cee | C:\Users\pony\clawd | 本机 18787 |
| erkang | cli_a92ccb1e69a29cc7 | C:\Users\pony\clawd-erkang | 本机 18787 |
| xiaoma | cli_a92cc46920395bc6 | C:\Users\pony\clawd-xiaoma | 本机 18787 |

### 关键发现

✅ **所有 Agent 都在同一台机器上**
- 通过 `agents.list` 配置为子 agent
- 共享同一个 Gateway (port 18787)
- 通过不同的飞书 bot 账号区分

❌ **如果分布在多台电脑**
- 每台电脑需要独立的 Gateway
- 需要网络可达配置
- 当前配置不支持跨机器调用

## 🔄 调用机制方案

### 方案 1：飞书群组消息（✅ 当前可用）

**原理**：
- 所有 Agent 加入同一个飞书群组
- 通过 @提及 触发对应 Agent
- 使用任务交接模板传递上下文

**优点**：
- ✅ 无需额外配置
- ✅ 天然支持跨机器
- ✅ 有完整的消息记录
- ✅ 支持人工介入

**缺点**：
- ❌ 依赖飞书平台
- ❌ 实时性稍差
- ❌ 需要人工@触发

**实施步骤**：
1. 创建飞书项目协作群
2. 邀请所有 Agent 机器人入群
3. 使用任务交接模板@下一环节负责人

**示例**：
```
【任务交接】
任务名称：环氧地坪 SKU 调研
当前环节：小康 → 尔康
交付物：https://feishu.cn/docx/xxx
完成情况：✅ 全部完成
@尔康 请开始策划方案
```

---

### 方案 2：sessions_spawn 子会话（✅ 同机器可用）

**原理**：
- 主 Agent 使用 `sessions_spawn` 创建子会话
- 子会话执行特定任务
- 任务完成后返回结果

**优点**：
- ✅ 编程式调用
- ✅ 实时响应
- ✅ 结果可直接使用
- ✅ 支持任务委派

**缺点**：
- ❌ 仅限同一 Gateway 内
- ❌ 不支持跨机器
- ❌ 子会话无独立飞书账号

**示例代码**：
```json
{
  "action": "sessions_spawn",
  "task": "分析环氧地坪市场数据",
  "agentId": "erkang",
  "mode": "run",
  "timeoutSeconds": 300
}
```

---

### 方案 3：Gateway 网络暴露（⚠️ 需配置）

**原理**：
- 将 Gateway 从 loopback 改为网络绑定
- 配置 auth token 认证
- 跨机器通过 HTTP API 调用

**当前配置**：
```json
"gateway": {
  "port": 18787,
  "mode": "local",
  "bind": "loopback",  // 需改为 "any"
  "auth": {
    "mode": "token",
    "token": "xxx"
  }
}
```

**修改方案**：
```json
"gateway": {
  "port": 18787,
  "mode": "local",
  "bind": "any",  // 允许外部访问
  "auth": {
    "mode": "token",
    "token": "共享 token"
  }
}
```

**优点**：
- ✅ 支持跨机器调用
- ✅ 编程式 API
- ✅ 实时响应

**缺点**：
- ⚠️ 需要网络安全配置
- ⚠️ 需要固定 IP 或内网穿透
- ⚠️ 有安全风险

**实施步骤**：
1. 修改 gateway.bind 为 "any"
2. 配置防火墙允许 18787 端口
3. 所有机器使用相同 token
4. 通过 HTTP API 调用

### 方案 4：飞书云文档触发（✅ 推荐）

**原理**：
- 创建共享任务看板（飞书多维表格）
- Agent 轮询任务状态变化
- 任务完成后更新状态

**任务看板结构**：
| 任务 ID | 任务名称 | 当前环节 | 负责人 | 状态 | 交付物 | 更新时间 |
|--------|----------|----------|--------|------|--------|----------|
| 001 | SKU 调研 | 尔康 | 小康 | 进行中 | [链接] | 2026-03-07 |

**Agent 轮询逻辑**：
```
每 5 分钟检查任务看板
IF 任务.负责人 == 我 AND 任务.状态 == "待处理"
  THEN 执行任务
  AND 更新任务.状态 = "进行中"
  AND 更新任务.更新时间 = NOW
```

**优点**：
- ✅ 天然支持跨机器
- ✅ 有完整的任务记录
- ✅ 支持人工查看和介入
- ✅ 不依赖网络暴露

**缺点**：
- ❌ 需要轮询（实时性稍差）
- ❌ 依赖飞书 API

---

### 方案 5：飞书日历事件触发（✅ 推荐）

**原理**：
- 创建共享日历
- 任务作为日历事件
- 事件开始时间触发 Agent 执行

**优点**：
- ✅ 支持定时任务
- ✅ 天然支持跨机器
- ✅ 有人工可见性

**缺点**：
- ❌ 适合计划性任务
- ❌ 不适合即时任务

## ⚠️ 安全注意事项

1. **不要暴露 Gateway 到公网**
   - 仅限内网访问
   - 配置强 token
   - 监控访问日志

2. **飞书机器人权限**
   - 最小权限原则
   - 定期审查权限
   - 不共享 appSecret

3. **任务交接验证**
   - 重要任务需要人工确认
   - 关键决策保留审批环节
   - 敏感操作需要二次确认
