### 1.3 现有工作流问题

1. **缺乏 Gate 门控机制** - 没有明确的阶段审核点
2. **Agent 调用靠人工** - 没有自动触发规则
3. **质量门禁缺失** - 交付标准不清晰
4. **知识沉淀不足** - 经验未形成组织资产
5. **技术深度不足** - 缺乏核心技术攻关角色
6. **设计能力薄弱** - UI/UX 无专人负责

### 2.2 IPD 9 Gate 门控流程（核心）

参考 IPD 实战案例，定义我们团队的 9 个 Gate：

```
┌─────────────────────────────────────────────────────────────────┐
│                      IPD 产品开发流程                            │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Gate 0        Gate 1        Gate 2        Gate 3              │
│  概念筛选  →   商业分析  →   用户验证  →   需求分析            │
│  (大管家)      (尔康)        (尔康)        (小马 + 大管家)       │
│     ↓             ↓             ↓             ↓                 │
│  Gate 4        Gate 5        Gate 6        Gate 7        Gate 8│
│  产品设计  →   UI 设计   →   研发编码  →   测试验收  →   发布上线│
│  (小马)        (小马)        (小马)        (测试 + 大管家) (大管家 + 小马)│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 各 Gate 详细说明

| Gate | 阶段 | 主导 Agent | 参与 Agent | 核心产出物 | 审核标准 |
|------|------|-----------|-----------|-----------|---------|
| **Gate 0** | 概念筛选 | 大管家 | 尔康、小康 | 市场规模分析、目标拆解、竞品分析、产品定价、财务模型、风险评估 | 市场规模>1 亿、ROI>30% |
| **Gate 1** | 商业分析 | 尔康 | 大管家、小康 | 执行摘要、TAM/SAM/SOM、商业模式画布、财务预测、风险评估 | 商业模式清晰、风险可控 |
| **Gate 2** | 用户验证 | 尔康 | 大管家、小康 | 用户画像 (3-5 类)、付费意向验证 (>60% 认可)、MVP 方案、验证指标 | 用户认可度>60% |
| **Gate 3** | 需求分析 | 小马 | 大管家、核心工程师 | 产品概述、用户角色、场景清单、功能需求 (P0/P1/P2)、页面需求 | 需求覆盖率>90% |
| **Gate 4** | 产品设计 | 小马 | 核心工程师、大管家 | 页面流程、功能 BDD、技术方案、风险评估、验收标准 | 技术方案可行、风险已识别 |
| **Gate 5** | UI 设计 | **设计师** | 小马、大管家 | UI 设计规范、各端 UI、交互原型、用户体验报告 | 符合设计规范、用户体验良好 |
| **Gate 6** | 研发编码 | **核心工程师** | 小马、技术专家 | 框架设计、数据库设计、API 文档、源码、代码 Review 报告 | 代码规范、文档完整、Review 通过 |
| **Gate 7** | 测试验收 | 测试 | 大管家、核心工程师、小马 | 测试报告、评分 (>8/10)、Bug 清单、通过率 | 评分>8 分、通过率>95% |
| **Gate 8** | 发布上线 | 大管家 | 核心工程师、小马 | 部署计划、上线检查清单、监控配置、应急预案 | 检查项 100% 完成 |

### 2.3 工作流自动触发机制

基于 OpenClaw 实战指南的关键词触发规则：

#### 触发词 → 工作流映射

| 用户提到关键词 | 触发工作流 | 主导 Agent | 自动动作 |
|---------------|-----------|-----------|---------|
| "项目"、"产品"、"开发"、"新功能" | IPD | 小马 | spawn 小马，创建项目卡片 |
| "战略"、"目标"、"规划"、"年度" | DSTE | 大管家 | spawn 大管家，启动规划流程 |
| "客户"、"销售"、"成交"、"线索" | LTC | 大管家 | spawn 大管家，协调销售相关 |
| "增长"、"营销"、"推广"、"活动" | MTL/MM | 尔康 | spawn 尔康，制定营销方案 |
| "测试"、"QA"、"验收"、"Bug" | PQA | 测试 | spawn 测试，启动测试流程 |
| "问题"、"投诉"、"反馈"、"工单" | ITR | 大管家 | spawn 大管家，协调问题解决 |
| "数据"、"分析"、"报告"、"趋势" | 数据大脑 | 小康 | spawn 小康，生成分析报告 |
| "技术"、"方案"、"架构"、"核心" | 技术支持 | **核心工程师** | spawn 核心工程师，输出技术方案 |
| "设计"、"UI"、"UX"、"界面"、"视觉" | UI 设计 | **设计师** | spawn 设计师，启动设计流程 |

#### 阶段自动触发规则

```yaml
# IPD 流程中的自动触发
Gate 0 完成 → 自动触发 Gate 1（尔康）
Gate 1 完成 → 自动触发 Gate 2（尔康）
Gate 2 完成 → 自动触发 Gate 3（小马）
Gate 3 完成 → 自动触发 Gate 4（小马）
Gate 4 完成 → 自动触发 Gate 5（设计师）
Gate 5 完成 → 自动触发 Gate 6（核心工程师）
Gate 6 完成 → 自动触发 Gate 7（测试 + 大管家）
Gate 7 完成 → 自动触发 Gate 8（大管家 + 核心工程师）

# 跨工作流触发
LTC 线索获取 → 触发 MTL 营销优化
ITR 问题超过 3 次 → 触发 PQA 质量改进
PQA 评分<8 分 → 触发 IPD 需求重新分析
```

### 2.4 质量门禁标准

参考 IPD 实战案例的评分机制：

#### Gate 审核评分表（满分 10 分）

| 维度 | 权重 | 评分标准 | 通过线 |
|------|------|---------|-------|
| **完整性** | 25% | 产出物是否齐全、文档是否完整 | ≥8 分 |
| **准确性** | 25% | 数据是否准确、分析是否有依据 | ≥8 分 |
| **可行性** | 20% | 方案是否可执行、资源是否到位 | ≥7 分 |
| **风险可控** | 15% | 风险是否识别、应对是否充分 | ≥7 分 |
| **创新性** | 15% | 是否有差异化、竞争力 | ≥6 分 |

**综合评分 ≥8 分** 方可进入下一阶段

#### 测试验收评分标准（参考 IPD 案例 9.5 分）

| 测试类型 | 覆盖率要求 | 通过率要求 |
|---------|-----------|-----------|
| 功能测试 | 100% P0 功能 | 100% |
| 性能测试 | 核心接口 | 响应<500ms |
| 兼容性测试 | 主流浏览器/设备 | 95% |
| 安全测试 | OWASP Top 10 | 无高危漏洞 |
| 用户体验 | 关键路径 | 无阻塞问题 |

## 四、关键成功因素

### 4.1 必须做到

1. **Gate 门控严格执行** - 不达标不允许进入下一阶段
2. **知识沉淀常态化** - 每个 Gate 完成后 24 小时内沉淀
3. **质量评分真实客观** - 不走过场，评分<8 分必须返工
4. **安全红线不触碰** - 高危操作必须人工确认

### 4.2 避免的坑

1. ❌ 为了进度跳过 Gate 审核
2. ❌ 知识沉淀流于形式
3. ❌ Agent 调用还是靠人工（未实现自动触发）
4. ❌ 测试验收自己测自己的代码
